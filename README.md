# AIBA 需求工程助手 / AIBA Requirements Engineering Assistant

> AI 驱动的需求工程全流程技能，将模糊的业务描述转化为开发团队可直接执行的结构化需求文档。
>
> An AI-powered, full-lifecycle requirements engineering skill that transforms vague business descriptions into structured, developer-ready requirement documents.

---

## 功能概览 / Features

本技能覆盖需求工程的 **10 个阶段**，从原始需求补全到验收用例生成，全流程自动化：

| 阶段 | Stage | 功能 | 产出 |
|------|-------|------|------|
| 1 | 原始需求优化 | 业务背景追问与补全 | 业务背景补全问答清单 |
| 2 | 需求分析 | 9 个子步骤：干系人、角色、术语、流程、场景、依赖、功能等 | 结构化分析文档（Markdown 表格 + PlantUML） |
| 3 | UI 生成 | 数据模型 → 表单 / 查询 / 大屏 / 交互页面 | 数据模型 + 可运行 HTML 原型 |
| 4 | 需求确认 | 场景故事线可视化 | 可视化确认页面（HTML） |
| 5 | 需求表达 | 渐进式工具选择（13 个工具按需执行） | 状态图、决策矩阵、校验矩阵、权限矩阵等 |
| 6 | 资源时间模型 | 日历视图与交互需求 | 日历视图文档 |
| 7 | 需求质量扫描 | 9 类质量问题检测 | 质量扫描报告 |
| 8 | 需求计划 | 拆分、估算、用户故事地图 | 需求拆分文档 + 估算结果 |
| 9 | 需求开发 | 变更影响分析（数据模型 / UI / 外部依赖） | 变更影响分析报告 |
| 10 | 需求验收 | 验收用例生成 | 验收测试用例 |

### 阶段 5：渐进式工具选择 / Progressive Tool Selection

阶段 5 是本技能的核心特色。包含 13 个专业工具，但并非每个项目都需要全部执行。AI 会通过**语义判断**自动选择最小合适工具集：

| 工具 | 用途 | 触发条件 |
|------|------|---------|
| 5.0 状态梳理 | 梳理业务对象生命周期状态 | 有明确状态变化（如草稿→审核→完成） |
| 5.1 状态图 | 生成状态流转可视化图 | 依赖 5.0 |
| 5.2 决策矩阵 | 梳理条件分支判断逻辑 | 流程中存在分支判断 |
| 5.3 表达式树 | 结构化复杂布尔逻辑 | 3 个以上条件的 AND/OR/XOR 组合 |
| 5.6 校验矩阵 | 字段级输入校验规则 | 有表单/导入/API 写入场景 |
| 5.9 判定矩阵 | 参数→结果映射结构化 | 有参数区间映射或多参数组合判定 |
| 5.10 后台校验 | 服务端校验与外部依赖 | 有增删改操作或外部系统调用 |
| 5.11 编号生成 | 业务单据自动编号规则 | 有需编号的业务单据实体 |
| 5.12 定时任务 | 定时/超时/到期自动处理 | 有定时触发需求 |
| 5.13 文件导入 | 批量数据导入方案 | 涉及 Excel 上传/批量导入 |
| 5.14 文件导出 | 数据导出或报表方案 | 涉及导出/报表/打印 |
| 5.15 角色权限矩阵 | 角色功能访问权限 | 2+ 角色且功能访问范围不同 |
| 5.16 数据权限 | 行级数据隔离规则 | 有"只能看自己的"等隔离需求 |

**判断原则**：宁可跳过也不要多选，每个工具独立判断，不确定则跳过。用户可随时覆盖 AI 的选择。

## 安装方法 / Installation

本技能基于 Claude Code 的 Skills 体系，兼容所有支持该体系的平台（如 Claude Code、OpenClaw 等）。

### 方式一：Git Clone（推荐）

```bash
# 克隆到技能目录
git clone https://github.com/AndyClouder/AIBA.git ~/.claude/skills/AIBA需求工程助手
```

### 方式二：手动安装

1. 下载本项目 ZIP 文件
2. 解压到技能目录：`~/.claude/skills/AIBA需求工程助手/`

> **注意**：确保目录名与 `SKILL.md` 中的 `name` 字段一致（`AIBA需求工程助手`）。

## 使用方法 / Usage

安装完成后，技能会自动根据用户输入触发。以下是几种典型用法：

### 完整流程 / Full Workflow

直接描述你的业务需求，AI 会从阶段 1 开始逐步推进：

```
我要做一个员工请假审批系统。员工提交请假申请，主管审批，HR备案。
```

### 单步执行 / Single Step

指定某个具体任务，AI 会直接跳到对应阶段：

```
帮我画个流程图
生成这个需求的数据模型
做干系人识别
```

### 继续会话 / Resume Session

如果 `docs/requirements/manifest.json` 已存在，AI 会自动检测进度并从上次中断处继续：

```
继续之前的需求分析
```

### 独立质量检查 / Standalone Quality Check

对已有需求进行质量扫描、拆分估算或验收用例生成：

```
对当前需求做质量扫描
帮我拆分和估算需求
生成验收用例
```

## 需求类型适配 / Requirement Type Adaptation

不同类型的业务需求会自动适配不同的分析深度。以下是 4 种典型场景的阶段 5 工具选择策略：

| 需求类型 | 示例 | 执行的工具 | 跳过的工具 |
|---------|------|-----------|-----------|
| 复杂审批类 | 采购合同审批（状态流转、多级审批、金额阈值） | 5.0 5.1 5.2 5.6 5.11 5.12 5.15 | 5.3 5.9 5.10 5.13 5.14 5.16 |
| 纯数据录入类 | 员工通讯录（CRUD、无流程） | 5.6 5.15 | 其余全部 |
| 混合型 | 考试成绩管理（流程 + 导入 + 权限 + 判定） | 5.0 5.1 5.2 5.6 5.9 5.10 5.11 5.13 5.14 5.15 5.16 | 5.3 5.12 |
| 极简类 | 内部公告（发布 + 查看） | 无 | 全部跳过 |

## 技术架构 / Architecture

### 渐进式披露 / Progressive Disclosure

本技能采用三层渐进式加载架构，避免一次性加载过多内容：

1. **SKILL.md**（始终在上下文中）— 流程控制、阶段路由、输出规范
2. **references/ 按需加载**（执行到对应阶段时读取）— 45 个专业提示词文档
3. **输出文件**（持久化到文件系统）— 跨会话恢复、变量传递

### 变量传递 / Variable Passing

各阶段之间通过 `manifest.json` 的 `variables_index` 实现变量传递。每个阶段完成后，将关键变量映射到对应文件路径，后续阶段通过 `Read` 工具加载。

## 测试验证 / Testing

`evals/` 目录包含 7 个测试用例，覆盖以下场景：

| 测试 ID | 场景 | 验证内容 |
|---------|------|---------|
| 1 | 请假审批系统（阶段 1） | 原始需求优化 + manifest 创建 |
| 2 | 请假审批系统（阶段 2 续） | 会话恢复 + 从阶段 2 继续 |
| 3 | 会议室预约（单步执行） | 直接跳到干系人识别子步骤 |
| 4 | 采购合同审批（阶段 5） | 复杂审批类的渐进式工具选择 |
| 5 | 员工通讯录（阶段 5） | 纯数据录入类的最小工具集 |
| 6 | 考试成绩管理（阶段 5） | 混合型的完整工具覆盖 |
| 7 | 内部公告（阶段 5） | 极简需求的全部跳过 |

其中测试 4-7 通过 40 条期望断言验证工具选择的准确性。
